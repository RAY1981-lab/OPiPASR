+# Техническое задание (MVP)
+
+## 1. Цель проекта
+Создать веб-систему поддержки принятия решений для РТП/РЛЧС/ЦУКС, которая:
+- принимает данные от ИБАС;
+- сохраняет и нормализует их в БД;
+- выполняет аналитическую/ИИ-обработку по алгоритмам из диссертации;
+- отображает прогнозы и рекомендации в веб-интерфейсе с журналом решений.
+
+## 2. Роли пользователей
+- **Оператор (ЦУКС/диспетчер)**: просмотр потока событий, подтверждение/отклонение рекомендаций.
+- **Руководитель (РТП/РЛЧС)**: просмотр обстановки, прогноза, сводки рисков и приоритетов.
+- **Администратор**: управление пользователями, справочниками, источниками данных.
+
+## 3. Основные сценарии
+1. ИБАС отправляет событие в систему.
+2. Сервис приёма валидирует и сохраняет «сырой» пакет.
+3. Пайплайн нормализации формирует доменную сущность инцидента.
+4. ИИ-модуль запускает прогноз и формирует рекомендации.
+5. Результаты отображаются в интерфейсе с объяснением.
+6. Оператор фиксирует действие (принято/отклонено/изменено).
+7. Все шаги пишутся в аудит и используются для последующего анализа качества.
+
+## 4. Функциональные требования
+
+### 4.1 Приём данных от ИБАС
+- Поддержка POST `/api/v1/ibas/events`.
+- Идемпотентность по внешнему ключу события (`source_event_id`).
+- Сохранение исходного payload без потерь.
+- Валидация обязательных полей: источник, время, тип события, координаты/локация (если есть).
+
+### 4.2 Нормализация и обогащение
+- Приведение типов событий к внутреннему справочнику.
+- Нормализация времени в UTC.
+- Геопривязка (при наличии координат).
+- Привязка к действующему инциденту либо создание нового.
+
+### 4.3 ИИ/аналитика
+- Запуск по событию и/или расписанию.
+- Выходные данные:
+  - прогноз (горизонт, вероятность, уровень риска);
+  - рекомендации (список действий с приоритетами);
+  - объяснение (ключевые факторы).
+- Фиксация версии модели и параметров запуска.
+
+### 4.4 UI/операторская панель
+- Дашборд инцидентов (статус, приоритет, обновления).
+- Карточка инцидента: события, прогноз, рекомендации, история действий.
+- Фильтрация по территории, времени, типу угрозы.
+
+### 4.5 Аудит и отчётность
+- Лог действий пользователей.
+- История изменений рекомендаций и прогнозов.
+- Экспорт отчётов по инциденту (CSV/JSON).
+
+## 5. Нефункциональные требования
+- **Надёжность**: graceful degradation при недоступности ИИ.
+- **Безопасность**: RBAC, журналирование, хранение секретов вне кода.
+- **Производительность**: приём события ≤ 500 мс (без тяжёлой аналитики).
+- **Наблюдаемость**: метрики, трассировка, структурированные логи.
+- **Масштабируемость**: разделение ingestion/analysis/API слоёв.
+
+## 6. API (черновой контракт)
+
+### 6.1 Ingestion
+- `POST /api/v1/ibas/events`
+- `GET /api/v1/ibas/events/{id}`
+
+### 6.2 Incidents
+- `GET /api/v1/incidents`
+- `GET /api/v1/incidents/{id}`
+
+### 6.3 AI results
+- `GET /api/v1/incidents/{id}/forecasts`
+- `GET /api/v1/incidents/{id}/recommendations`
+- `POST /api/v1/incidents/{id}/recommendations/{rid}/decision`
+
+## 7. Этапы MVP
+1. Приём данных + БД + базовый API.
+2. Интерфейс дашборда и карточки инцидента.
+3. Интеграция ИИ-алгоритмов из диссертации.
+4. Метрики качества прогноза и операционное сопровождение.
+
+## 8. Критерии приёмки MVP
+- Система принимает и сохраняет события от ИБАС.
+- Создаётся и обновляется карточка инцидента.
+- Для инцидента доступны хотя бы один прогноз и набор рекомендаций.
+- Решения операторов сохраняются и доступны в журнале.
+- Доступ разграничен минимум по трём ролям.
